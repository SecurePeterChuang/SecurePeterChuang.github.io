安全設計原則與模型有哪些? 如何用?

安全設計原則 (Secure Design Principals)

Domain 3.1, 2o24o4原廠簡報 Domain p.3 ~ 17

  1. 威脅建模 Threat Modeling：用攻擊者的角度來識別、分析系統中潛在的安全威脅。

    我們曾在 Domain 1 提及 STRIDE threat modeling <Domain 1 p.82>，Domain 4 將會提及 Cyber-security Kill Chain <Domain 4 p.14>。

  2. 最小特權 Least Privilege：從系統面給予使用者完成工作的最小權限。
  3. 縱深防禦 Defense in Depth, DiD：多層防禦策略，結合多種安全控制措施保護系統。加密是資安防護的最後一道防線。
  4. 安全默認設置 Secure Defaults：系統應用程式的預設配置應設定為最安全的狀態，以減少暴露於威脅的風險。ex. Wi-Fi AP 原廠初始設置早期可能沒有密碼或該廠共用初設密碼，這就是一個沒有 Secure Defaults 的實作。

    202404版原廠簡報拿掉這塊，所以你找不到

  5. 安全失效 Fail Securely：系統失效時，也能保持安全狀態(Security)，防止敏感資料洩漏或其他安全問題，ex. 網站出現404錯誤時，不應洩漏除錯(Debug)資訊。 vs. Fail Safe (一樣翻譯為安全失效)，指系統失效時，系統轉向一個安全狀態以避免隊人員造成傷害，ex. 門禁系統與消防受信主機連線，偵測到失火即關閉門禁管制。
  6. 職責分離 Segregation of Duties, SoD：把重要的職責和任務分散給多個人或團隊中，以降低有意或無意造成的風險。ex. 請購、採購、會計、出納流程。
  7. 保持簡單小巧 Keep it simple and small：在設計系統保持簡潔與精簡，減少潛在的安全漏洞和錯誤。ex. 軟體的模組化設計，開發人員只需要關注小的、單一功能的模組或元件。
  8. 零信任 Zero trust：一種不信任任何內部或外部資源的安全模型，所有資源都須經過嚴格的驗證和授權。

    推薦閱讀：【回歸根本定義，捍衛網路戰爭的無形防線】零信任之父第一手導讀零信任架構

  9. 信任但驗證 Trust, but Verify：在信任某一資源(或實體)時，仍進行必要的驗證作業以確認安全性。
  10. 隱私設計 Privacy by design：在設計產品或系統時，要將隱私保護納入考量(特別是合規性)，確保全流程都考慮隱私保護。

    推薦閱讀：台灣數位身分證導入失敗案例

  11. 責任共擔 Shared responsibility：在雲服務中使用，指服務供應商和使用者在安全管理上共同承擔責任，根據不同的服務模型(Service Model)分配不同的安全責任。

    推薦閱讀： Shared responsibility in the cloud

  12. 安全訪問服務邊緣 Secure access service edge, SASE：一種將網路安全功能和廣域網路(WAN)連結連接結合在邊緣，便於在分散式環境實施一致的安全政策。ex. 把公司內部的 VPN 或 FW 等傳統安全服務移到雲端，通過 SASE，員工無論在何處都可以透過 SASE 進行實時的安全檢查、身分驗證和權限管理，確保一致的安全政策保護。

    此為 202404 新增範圍

補充說明：「零信任」和「信任但驗證」的差異

以實體安全為例，當你通過門禁管制(可能是通關機或門禁卡機)驗證後，你可以在內部環境漫遊(偵查)，這即是信任但驗證的案例。

當一個區域網路(LAN)環境沒有進行區隔，無額外的內部防火牆或內部邊界設置，使用者進入內網後，在網路層級使用者可以自由選取系統進行訪問(敏感資源在系統層級仍會有進一步的驗證機制)，即是信任但驗證的案例。

在零信任的環境中，每一個行為(無論是來自內部還是外部)都被視為潛在的風險，這意味著所有的流量、設備和用戶，都必須被持續監控和驗證。

安全模型 (Security Models)

Domain 3.2, 2o24o4原廠簡報 Domain p.18 ~ 24，考試大綱提到 Star Model 其應指的是 BLP 和 Biba 所提到的 Star property。

課堂間的 Subject 和 Object 的簡單關係在這邊不再提。

  1. Bell-LaPadula (BLP)：專注於機密性，有三種安全性層次
    • Simple Security Property (簡單安全屬性)：No Read Up 不能向上讀取 (窺探機密)
    • Star Security Property (*(星)安全屬性)：No Write Down 不能向下寫入(洩漏機密)
    • Strong Star Security Property (粗體星安全屬性)，只能寫同級(嚴格控制資訊流)
  2. Biba：專注於完整性(可用數字精度類比)，也有三種安全性層次
    • Simple Integrity Property (簡單完整性屬性)，No Read Down 下能向下讀 (避免從低完整性的物件取得不精確的資料)
    • Star Integrity Property (*(星)完整性屬性)，No Write Up 不能向上寫 (破壞上層完整性)
    • Invocation Property (調用屬性)，低完整性的系統不能呼叫 (send service request) 到高完整性的系統
  3. Brewer and Nash (又稱 Chinese Wall 中國牆模型)：專注於解決利益衝突，特別使用於保護商業機密的場景。ex. 阻止使用者同時可訪問彼此有競爭關係的客戶資料，防止利益衝突。可透過標示 Subject 的 Clearance (利益衝突群組) 與 Object 的 Categorization 達成 (Subject 與 Object 的分級與分類請回去參考 Domain 2)。
  4. Clark-Wilson：專注於完整性，當 Subject 向 Object 提出交易時由另一方 (Program) 評估，是一種職責分離(SoD)的概念。主要達成三個目標
    • 避免未經授權的 Subject 進行變更，同 Biba。
    • 避免有經授權的 Subject 進行不良的變更，如帳戶餘額 10 元，不接受提款 100 元。
    • 維持系統的一致性，Subject 無法跳過 Program 直接從取 Object，確保套用一致的政策。
  5. Graham-Denning：專注於 Subject 與 Object 的八項基本操作，確保所有操作確保安全，包括 Subject 或 Object 的建立、刪除，向對象授予權限、轉移權限、取消權限與授予讀取權限。
  6. Harrison, Ruzzo, Ulman (HRU)：與Graham-Denning類似，但Subject 只能對 Object 進行有限的操作。(預先定義好的，沒有調整的空間)

安全架構 (Security Architecture) 與信息系統的安全功能

Domain 3.4

模型 Model：幫助我們將複雜的系統分解為組件的簡化表示，一旦我們了解組成複雜系統的組件，我們就可以保護每個組件，從而解決高度複雜系統內各方面的問題。這裡的安全模型 (Security Model) 指關注機密性、完整性與防止利益衝突的模型。

  1. State Machine model 狀態機模型(Textbook 有出現，但不在簡報中)：常用在軟體程序和邏輯電路中，由一組狀態、轉換規則以及輸入和輸出組成，每當狀態機收到輸入時，它會根據規則改變狀態，並依當前狀態產生輸出。與前一截的內容對應如下：
    • BLP model (基於狀態機的機密性模型)
    • Biba model (基於狀態機的完整性模型)
    • Brewer and Nash (依賴狀態機確認資料流動與規則)
    • Clark-Wilson Model (依賴狀態機強制執行交易的完整性約束)
    • Graham-Denning (用狀態機模型描述系統操作與權限管理)
    • Harrison, Ruzzo, Ulman (HRU) (用狀態機模型描述系統操作與權限管理)。
  2. Non-interference model 非干擾模型(Textbook 有出現，但不在簡報中)：常用在關注於機密性模型，高機密等級的 Subject 進行任何的操作都不會被低機密等級 Subject 觀察到，從而防止機密信息洩漏。
  3. Ring Model 環模型(202404簡報 p.35)：常用於作業系統 (OS) 的內核 (Kernel) 設計，其將系統操作分成不同層級的特權環，從 Ring 0 ~ Ring 3，Ring 0 有最高的特權，若出錯會造成死亡之藍屏 (Blue Screen of Death)，常見的作業系統沒有使用 Ring 1 和 2，Ring 3 是應用程序運作的地方，若出錯不會影響系統內核的運作。Ring 0 為 Kernel model，Ring 1 ~ 3 為 User Mode。

    推薦閱讀：2024.07.19 CrowdStrike 軟體更新導致系統崩潰的案例。

架構 Architecture：是指系統結構與其行為的高階描述，可確定各個組件與它們之間的相互關係與協作關係。如資訊系統架構(軟體架構、網路架構)、建築架構、組織架構(組織結構、業務流程的藍圖等)。

信息系統在架構上的安全功能

Domain 3.5

Must have 必要功能

  1. Privileged (model) instruction 特權模式指令：作業系統的操作模式分為 User Mode 和 Kernel Model，特權指令指的是只能在 Kernel Mode 下才能執行的指令。CPU的某些指令稍微有錯誤操作可能導致系統崩潰，這些指令被歸類為特權指令，需要使用系統呼叫 (System call) 來執行， ex. Clear memory, Write intro monitor memory…；而使用者平常在使用電腦時只能在 User Mode 下操作，無法使用在 Kernel Mode 下才能執行的特權指令，這樣可以避免系統崩潰的機會。
  2. Processor states 處理器狀態：在 CPU 運行的過程中處在的不同工作模式，這些模式決定 CPU 能夠執行哪些指令，確保用戶程序不會影響系統核心的運作。其分為兩個模式 Supervisor state 執行 Kernel mode 指令、Problem state 執行 User model 指令。
  3. Memory management 內存(記憶體)管理：目的為高效率使用內存資源、保護程序之間的內存、防止內存洩漏。
  4. Abstraction Layers 抽象層：隱藏系統複雜性並提供簡化的介面。ex. 銀行提款機只露出讀卡機、鍵盤、螢幕與網路介面，安裝工程師不用知道內部實作細節，這可增加系統的可移植性與安全性。
  5. Data and code space isolation / Code and data segregation 資料與代碼隔離：將資料與代碼儲存在不同記憶體空間中，防止惡意代碼執行攻擊。ex. 資料被當作可執行代碼運作。
  6. File systems attributes 檔案系統屬性：檔案系統屬性是用於描述檔案的元數據 (Meta data)，如唯讀、隱藏、系統存取權限等 (Windows 系統隨便一個檔案按右鍵可以看到)。這些屬性可以幫助控制檔案的訪問權限與其他操作行為。
  7. Security kernel 安全核心：Ring model 提過，SK 負責實施底層的安全策略和機制，維持系統整體安全性的核心部分。

Vital extensions 重要的延伸(nice to have)

  1. Access control 存取控制：一種安全管理機制，用於管理和限制使用者對資源的存取許可，Domain 5 會詳細的說明，先把幾個關鍵字記起來 MAC、DAC、NDAC。
  2. Virtual machines (VM) 虛擬機：一種軟體模擬環境，模擬實際電腦系統的功能，允許在一台實體的主機上同時運作多個操作系統，隔離各個系統與應用程式，從而提供資源利用率和提供測試及開發環境的靈活性。Domain 3.5 等下會介紹 VM 的細節。
  3. Sandbox environment 沙箱環境：一種隔離的測試和運行環境，允許在不影響主系統的情況下執行和測試未經檢驗的程序，可保護系統免受潛在惡意代碼攻擊。
  4. Cryptographic systems 密碼系統：一種使用加密、解密技術保障信息的機密性、完整性、真實性、不可否認性的技術。等下 Domain 3.6 & 3.7 詳細介紹。
  5. Trusted Platform Module (TPM) 可信平台模組：一種硬體安全模組 (Hardware Security Module, HSM)，實體上是一個專用的安全晶片，通常整合在主機板上，提供安全功能，如安全開機、金鑰生成/管理、設備身分驗證等。

補充說明：TPM 的安全功能

  1. 加密金鑰管理：TPM可以安全的產生、儲存和管理加密金鑰，確保這些金鑰不會被未授權的應用程式或使用者訪問。
  2. 平台完整性檢查 (防竄改保護)：TPM可以檢察系統啟動過程中的關鍵部件，保證系統從可信狀態啟動，防止系統被竄改。
  3. 設備身分驗證：設備可利用儲存在晶片中的金鑰進行身分驗證
  4. 資料保護：TPM可以加密儲存在系統中的敏感資料，ex. BitLokcer 實現整個硬碟的加密。

TPM 是實體上獨立的模組，在系統進入 User mode 前啟動，亦有資料燒錄至晶片上無法竄改等特性，使得 TPM 成為許多高安全性應用的基礎，在 Windows、Linux 中廣泛使用。

認識市面上的系統與其安全問題

Domain 3.5, 2o24o4原廠簡報 Domain p.39 ~ 42

身為資訊安全專家，在了解安全設計原則與模型後，你還需要知道市面上常見的系統有哪些? 各是什麼? 有什麼弱點? 有什麼減緩措施?

以下是 ISC2 認證考試大綱 擷取出來的所有需要認識的系統，我依說明的方便性做了順序的調整。

Client-based System 客戶端系統

  1. 介紹：大量存在組織中，不斷有新的系統加入、舊的系統退役。
  2. 弱點：繼承 (Inherent) 硬體、作業系統和應用程式軟體的弱點。(寄人籬下，屋頂漏水，你房間也跟著漏水)
  3. 減緩措施：
    • 在系統上安裝反惡意軟體及入侵偵測與預防軟體 (Host-based anti-malware and intrusion detection and prevention software)

      202404 原廠簡報 Domain 7 p.66

    • 安裝終端管理系統 (Endpoint management system) 可以掌握大量的客戶端狀況、推送軟體或指令、系統更新或修補的版本。當結合移動裝置管理 (Mobile Device Management, MDM) 實，可以降低裝置被偷或遺失的風險。
    • 使用者必須了解安全規定並安全地使用客戶端系統，資安意識教育訓練也是重要的一環。

      202404 原廠簡報 Domain 1 p.94

Server-based System 服務器端系統

  1. 介紹：為特定目的提供服務的系統 (ex. DNS、Web Server)。通常會集中管理、控制與維護。
  2. 弱點：
    • 繼承 (Inherent) 硬體、作業系統和應用程式軟體的弱點。
    • 連接到一個或多個可能會被網路攻擊的環境下
  3. 減緩措施：
    • 增加冗餘 (Redundant)，降低單點失效 (Single Point of Failure, SPoF) 導致可用性風險
    • 實體 (physical), 邏輯 (logical), 或管理 (administrative) 存取控制 (access control)

Database System 數據庫系統

  1. 介紹：圍繞著 Database management system (DBMS) 資料庫管理系統以及託管開系統的處理器與儲存系統而建置。
  2. 弱點：
    • 格式錯誤 (畸形) 的輸入 Malformed input，ex. 提款金額為正整數欄位，使用者卻輸入負數、含小數點的數字或大寫中文。
    • 拒絕服務 Denial of Service (DoS)，ex. 用大量的查詢請求癱瘓資料庫
    • 避免或繞過資料庫的存取控制機制 Avoiding or sidestepping access control，ex. 對暫存資料表的存取
  3. 減緩措施：輸入驗證、限制不正常流量、實施嚴格的存取控制政策

Industrial Control System (ICS) 工業控制系統

  1. 介紹：常見於工業用系統和關鍵基礎設施 (Critical Infrastructure, CI) 中，由簡單的電腦進行監控與控制。主要由下列三個組件構成

    • Supervisory Control and Data Acquisition (SCADA)：用於監控和控制分散在各地的設備，通過數據通信網絡連接到中央系統。位置：工廠中控室
    • Distributed Control Systems (DCSs)：將控制功能分配到多個自治的但相互通信的子單元。位置：產線控制
    • Programmable Logic Controllers (PLCs)：一種工業電腦，可用軟體編程，用於控制製造過程中的機械設備或自動化流程。位置：機械設備旁

  2. 弱點：

    • Limited functionality 有限的功能：ICS 專注於特定的工業功能，無法快速適應新出現的威脅
    • Limited protection 有限的保護：傳統 ICS 通常在資訊安全未被充分考慮的年代設計的，通常缺乏保護
    • Long, economically useful service life 壽命長且經濟實用：ICS 設備使用壽命長，設計到佈署可能超過數十年。期間因技術更新，設備可能未能採用最新的安全更新和防護技術。
    • Susceptible to misconfiguration and misuse 易發生錯誤配置與誤用：系統的複雜性高或使用者缺乏安全意識。
    • Highly susceptible to denial of service (DoS) attacks 極易受到 DoS 攻擊：ICS 對於 DoS 攻擊沒有太多抵抗力。
    • Attacks can produce physical effects 攻擊可能導致實體效果：網路攻擊直接影響設備，導致隊人員安全、環境或基礎設施造成實際損害。
    • Often unattended in remote locations 常在偏遠無人看守地區：缺乏監控，難以即時發現並回應安全事故。

  3. 減緩措施：

    • Isolated network infrastructure 隔離的網路基礎設施：氣隙網路 (Air-gap network)、防火牆 (Firewall)
    • Robust network connection restrictions, access control, and monitoring 強大的網路連線限制、存取控制與監控：實施嚴格的網絡連接限制和訪問控制措施，以確保只有授權用戶和設備能夠訪問 ICS。
    • Highly segmented or microsegmented networks 高度分段或微分割網路：網絡劃分為更小的區段或子網（即微分割），限制不同區段之間的通信流量。
    • Protected communication channels 受保護的通訊管道：使用實體保護(physical proteciton) 或加密技術（如 TLS 或 VPN）保護 ICS 的數據通信，以防止攔截和竄改。
    • Robust configuration control 強大的配置控制：制定嚴格的配置管理流程，確保系統所有變更都經過適當的審核和授權，並保持詳盡的審計記錄。

      TLS、VPN、Firewall 等技術會在 Domain 4 說明。

    實體保護將在 Domain 3.8 與 3.9 說明。

    Configuration management 會在 Domain 7 說明。

Embedded System 嵌入式系統

  1. 介紹：直接融入機械、電器、液壓系統或其他類型設備的微型電腦技術。
    • 與 PLC 的差異在於，PLC 是用於工業環境的設備，主要用於自動化機械的控制，由 sensor 的輸入訊號來控制其他部件。而嵌入式系統是具有專門功能的計算系統，用於家用電器、醫療設備等。
  2. 弱點：
    • Programming errors 編程錯誤
    • Web-based vulnerability 基於 Web 的漏洞
    • Weak access control or authentication 薄弱的存取控制或身分驗證
    • Poor cryptography practices 不好的密碼學實踐
    • Reverse engineering 逆向工程
    • Malware 惡意軟體
    • Eavesdropping 竊聽
